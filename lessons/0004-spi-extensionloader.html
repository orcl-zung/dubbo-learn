第 4 课 · 阶段二

钻进 ExtensionLoader + 自己写一个 SPI 扩展

第 3 课你拿到了地图和钥匙——这节课你亲自拧钥匙。先写一个自己的 SPI 扩展跑起来（改个 key 就生效），再钻进 ExtensionLoader 看它背后怎么「加载 → 注入 → 包装」，最后揭开 @Adaptive 在运行时编译出的那个类到底长什么样。

⏱ 约 25–35 分钟 ← 第 3 课 术语表 架构速查
本课的 tangible win

启动 Consumer，控制台冒出：

[lea-SPI] FirstLoadBalance 实例化 —— getExtension("first") 命中了我
RPC response result: Hello dubbo

你一行 Dubbo 框架代码都没改，就给它插了一个全新的负载均衡实现。这就是 SPI「可插拔」最直白的证明——而这节课要让你既会做、又懂为什么。

Part A · 主线技能：写一个自定义 LoadBalance

回忆第 3 课那张 LoadBalance 的 SPI 配置文件（random=...、roundrobin=...）。今天你在它旁边加一行你自己的，然后用一个 key 把它选中。

第 1 步：实现接口（我已铺进你的工程）

我在 dubbo-demo-consumer 里建好了 com.lea.consumer.loadbalance.FirstLoadBalance。它 implements LoadBalance，select 永远返回 invokers.get(0)——策略故意做到最简，让你盯住 SPI 本身：

package com.lea.consumer.loadbalance;

import org.apache.dubbo.rpc.Invocation;
import org.apache.dubbo.rpc.Invoker;
import org.apache.dubbo.rpc.cluster.LoadBalance;
import java.util.List;

public class FirstLoadBalance implements LoadBalance {

    public FirstLoadBalance() {
        // 被加载/实例化的信号（单 provider 也会打）
        System.out.println("[lea-SPI] FirstLoadBalance 实例化 —— getExtension(\"first\") 命中了我");
    }

    @Override
    public <T> Invoker<T> select(List<Invoker<T>> invokers, URL url, Invocation invocation) {
        // 真正参与「挑选」的信号（需要 ≥2 个 provider 才会被调到）
        System.out.println("[lea-SPI] FirstLoadBalance.select 被调用，候选 invoker 数 = " + invokers.size());
        return invokers.get(0);   // 永远挑第一个（演示用）
    }
}

留意源码里那个接口——它就是第 3 课见过的 LoadBalance：

@SPI(RandomLoadBalance.NAME)            // 默认实现是 "random"
public interface LoadBalance {
    @Adaptive("loadbalance")              // 这行是 Part C 的主角，先记住
    <T> Invoker<T> select(List<Invoker<T>> invokers, URL url, Invocation invocation);
}
不用加任何依赖

LoadBalance 接口随 dubbo-spring-boot-starter 传递进来了（starter → config-api → dubbo-cluster），所以你直接 implements 就能编译。

第 2 步：注册（我已建好 SPI 配置文件）

我也建好了配置文件 dubbo-demo-consumer/src/main/resources/META-INF/dubbo/org.apache.dubbo.rpc.cluster.LoadBalance：

# 文件名 = 扩展接口的全限定名
# 内容   = key=实现类全限定名
first=com.lea.consumer.loadbalance.FirstLoadBalance

这就是第 3 课讲的约定：文件名是接口全限定名，内容是 key=实现类。你给这个实现起个 key 叫 first，下一步用它把实现选出来。

第 3 步：选中 —— 这一步你来（全课的钥匙转动）

打开 ConsumerApplication，把 @DubboReference 改成：

@DubboReference(loadbalance = "first")   // ← 用 key "first" 选你刚注册的实现
private DemoService demoService;

就这一行。这一步是整节课的「点睛」——你用一个 key，把 Dubbo 集群层要用的负载均衡换成了你自己写的那个，框架代码一行没碰。

第 4 步：跑

# Nacos 先起来，然后：
# 终端 1：Provider
cd dubbo-learn-practice/dubbo-demo-provider && mvn spring-boot:run
# 终端 2：Consumer
cd dubbo-learn-practice/dubbo-demo-consumer && mvn spring-boot:run

Consumer 控制台会打印：

[lea-SPI] FirstLoadBalance 实例化 —— getExtension("first") 命中了我
RPC response result: Hello dubbo
为什么只看到「实例化」，没看到「select 被调用」？

这是本课最容易被绊倒的地方，也是最有价值的发现。集群层 AbstractClusterInvoker.doSelect 对「只有 1 个 provider」会短路：

if (invokers.size() == 1) {
    return invokers.get(0);        // ← 单 provider，直接返回，根本不调 loadbalance.select
}

所以单 provider 时你的 select() 不会被调。但同一文件的 invoke() 每次调用都先跑 initLoadBalance(...)，它会 getExtension("first") 把你的实现按 key 加载出来——所以构造器日志一定会打，这已经足以证明「我的扩展被选中了」。

想看 select() 真正跑？起第 2 个 Provider，覆盖端口即可：

# 再起一个 Provider 实例（不同协议端口 + 不同 server 端口）
mvn spring-boot:run -Ddubbo.protocol.port=20882 -Dserver.port=9002

两个 Provider 都注册到 Nacos 同一接口下 → Consumer 看到 2 个 invoker → select() 被调，而且因为「永远挑第一个」，流量只打向其中一个。

闭环到源码入口

这一切的源头，就是 AbstractClusterInvoker.initLoadBalance 里这一句——它是「按 key 取扩展」的运行时入口：

return applicationModel
    .getExtensionLoader(LoadBalance.class)
    .getExtension(... LOADBALANCE_KEY, DEFAULT_LOADBALANCE));   // 读 URL 的 loadbalance 参数，默认 random

你配了 loadbalance=first，它就读到 "first"，去取你的 FirstLoadBalance。下一节，我们钻进这条 getExtension("first") 背后到底发生了什么。

Part B · 读源码：getExtension("first") 怎么变成你的类

3.x 关键变化（别被 2.x 教程带偏）

网上多数 Dubbo SPI 教程写的入口是 ExtensionLoader.getExtensionLoader(Class)——这个静态方法在 3.x 已 @Deprecated。3.x 的正经入口是 scopeModel.getExtensionLoader(type)：每个 SPI 接口对应一个 ExtensionLoader，且绑定到 ScopeModel（应用模型）。这是 3.x 能「在一个进程里跑多个独立 Dubbo 应用」的基础。下面生成 $Adaptive 类时，用的也是 scopeModel.getExtensionLoader(...)。

getExtension("first") 命中缓存就直接返回；首次访问会走到 createExtension。把它精简到只剩骨干（真实源码在 dubbo-common/.../ExtensionLoader.java）：

private T createExtension(String name, boolean wrap) {
    // ① 按名找类（首次访问时触发：读所有 SPI 配置文件，建出 name→Class 的表）
    Class<?> clazz = getExtensionClasses().get(name);

    T instance = extensionInstances.get(clazz);          // 反射 new（单例，按 Class 缓存）
    if (instance == null) {
        extensionInstances.putIfAbsent(clazz, createExtensionInstance(clazz));
        instance = extensionInstances.get(clazz);

        injectExtension(instance);                        // ② IOC：扫 setter，注入其它扩展
    }

    if (wrap) {                                           // ③ AOP：用 Wrapper 逐层包起来
        for (Class<?> wc : sorted(cachedWrapperClasses)) {
            instance = wc.getConstructor(type).newInstance(instance);
            injectExtension(instance);
        }
    }
    return instance;
}

三步看清「一个扩展被造出来」的全过程：

① 找类：getExtensionClasses()

首次调用时触发 loadExtensionClasses()，它遍历三个 LoadingStrategy 目录，读出每个目录下「文件名 = 接口全限定名」的配置文件，解析 key=类，建出一张 name → Class 的表，之后就一直用缓存。

目录（LoadingStrategy）放什么你的扩展放哪
META-INF/dubbo/给业务/第三方用的扩展这里（你的 FirstLoadBalance）
META-INF/dubbo/internal/Dubbo 自带的内部扩展—
META-INF/services/兼容 Java 原生 SPI—

② IOC 注入：injectExtension(instance)

遍历实例的所有 setter（setX、单参、非基本类型、未标 @DisableInject），对每个 setter：injector.getInstance(参数类型, 属性名) 拿到要注入的扩展，再 method.invoke。这就是第 3 课说的「扩展里写个 setter，框架自动帮你注入别的扩展」的源码落地。

③ Wrapper 包装（AOP）：cachedWrapperClasses

把缓存里的 Wrapper 类逐层 new Wrapper(instance) 套到你的实例外面。Filter 链就是这么来的——Part D 你会亲眼见到它的运行时体现。

一个类进哪个桶？loadClass 的三分法

读配置文件时，loadClass 把每个类分进三个桶之一：

判定条件进哪个桶典型例子
类上标了 @Adaptiveadaptive 桶（cachedAdaptiveClass）很少手写，多数靠运行时编译（见 Part C）
有「只接收 SPI 接口类型参数」的构造器Wrapper 桶（cachedWrapperClasses）ProtocolFilterWrapper
其余普通扩展桶（name→Class 表）你的 FirstLoadBalance、RandomLoadBalance
判定 Wrapper 只看一件事

一个类是不是 Wrapper，Dubbo 不看注解、不看名字，只看有没有一个构造器：参数恰好只有一个、且类型就是 SPI 接口本身（isWrapperClass）。有，就当 Wrapper——它会被用来包住别的扩展。

Part C · @Adaptive 自适应：运行时编译出的那个类长什么样

回到 LoadBalance.select 上那个 @Adaptive("loadbalance")。框架在编译期根本不知道你运行时要哪个负载均衡，所以它的做法是：在运行时现写、现编译一个「门面类」，这个门面从 URL 读 loadbalance 参数，再 getExtension(extName) 拿真正的实现。

下面是 Dubbo 为 LoadBalance 生成的那个类（按源码 AdaptiveClassCodeGenerator 精确还原）——这就是第 3 课欠你的「@Adaptive 编译出什么类」的兑现：

package org.apache.dubbo.rpc.cluster;
import org.apache.dubbo.rpc.model.ScopeModel;
import org.apache.dubbo.rpc.model.ScopeModelUtil;

public class LoadBalance$Adaptive implements org.apache.dubbo.rpc.cluster.LoadBalance {

    public org.apache.dubbo.rpc.Invoker select(java.util.List arg0,
            org.apache.dubbo.common.URL arg1, org.apache.dubbo.rpc.Invocation arg2)
            throws org.apache.dubbo.rpc.RpcException {

        if (arg1 == null) throw new IllegalArgumentException("url == null");
        org.apache.dubbo.common.URL url = arg1;
        if (arg2 == null) throw new IllegalArgumentException("invocation == null");
        String methodName = arg2.getMethodName();

        // ★ 从 URL 读 loadbalance 参数；你配了 "first" 就是 first，没配就回落默认 "random"
        String extName = url.getMethodParameter(methodName, "loadbalance", "random");
        if (extName == null) throw new IllegalStateException("Failed to get extension name from url ...");

        org.apache.dubbo.rpc.model.ScopeModel scopeModel =
                ScopeModelUtil.getOrDefault(url.getScopeModel(), org.apache.dubbo.rpc.cluster.LoadBalance.class);

        // ★ 按取到的名字，去 ExtensionLoader 拿真正的实现（就是 Part B 钻的那条路）
        org.apache.dubbo.rpc.cluster.LoadBalance extension =
                (org.apache.dubbo.rpc.cluster.LoadBalance) scopeModel
                        .getExtensionLoader(org.apache.dubbo.rpc.cluster.LoadBalance.class)
                        .getExtension(extName);

        return extension.select(arg0, arg1, arg2);
    }
}
看这两行，整节课就通了

String extName = url.getMethodParameter(methodName, "loadbalance", "random");——你配什么 key，这里就取出什么；下一句 getExtension(extName)——按名字去 Part B 那条流水线取真实现。这两行就是「URL 总线 + @Adaptive = 运行时按参数选实现」的源码本体。你在 Part A 配的 loadbalance=first，就是顺着这条链落到 FirstLoadBalance 上的。

又一个 3.x 细节

2.x 生成的门面里是 ExtensionLoader.getExtensionLoader(type) 这种静态调用；3.x 改成了 scopeModel.getExtensionLoader(...)——和 Part B 开头那条 ScopeModel 的变化同源。想亲眼看这段生成代码？把断点打在 AdaptiveClassCodeGenerator.generate()，能看到它拼出的整段源码字符串。

Part D · 加餐技能：写一个 Filter，看 SPI 的 Wrapper/AOP 面

LoadBalance 是「按 key 选一个」。Filter 展示 SPI 的另一面：多个 Filter 串成链（Wrapper 装饰），而且自动激活（@Activate，不用在 @DubboReference 里点名）。

这次轮到你亲手敲（把 SPI 从「看懂」变成「会写」）。在 dubbo-demo-consumer 新建 com.lea.consumer.filter.TimingFilter：

package com.lea.consumer.filter;

import org.apache.dubbo.common.constants.CommonConstants;
import org.apache.dubbo.rpc.Filter;
import org.apache.dubbo.rpc.Invocation;
import org.apache.dubbo.rpc.Invoker;
import org.apache.dubbo.rpc.Result;
import org.apache.dubbo.rpc.RpcException;

// group = CONSUMER：对所有消费者调用自动生效，无需在任何 @DubboReference 上点名
@Activate(group = CommonConstants.CONSUMER)
public class TimingFilter implements Filter {

    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        long start = System.currentTimeMillis();
        System.out.println("[lea-SPI] TimingFilter: 调用前 -> " + invocation.getMethodName());
        try {
            return invoker.invoke(invocation);   // 交给下一个 Filter / 真正的 Invoker
        } finally {
            System.out.println("[lea-SPI] TimingFilter: 调用后 <- 耗时 "
                    + (System.currentTimeMillis() - start) + "ms");
        }
    }
}

再注册它（新建配置文件 META-INF/dubbo/org.apache.dubbo.rpc.Filter）：

timing=com.lea.consumer.filter.TimingFilter

跑 Consumer，每次调用前后都会打印耗时。注意一个有意思的对照：Filter.invoke 没有「单 provider 短路」——所以哪怕你只有 1 个 Provider，它每次调用都跑（不像 LoadBalance.select 那样被短路）。这正说明 Filter 和 LoadBalance 在调用链上处在不同位置、职责不同。

Dubbo3 新东西（mission 点名要会）

3.0 把 Filter 拆成了两种：

  • ClusterFilter（3.0 新引入）：在负载均衡之前拦截，针对整个集群；配置文件是 META-INF/dubbo/org.apache.dubbo.rpc.cluster.filter.ClusterFilter。
  • Filter（你上面写的）：在负载均衡选定具体 Provider 之后拦截，绑定到单个 Provider 实例；配置文件是 META-INF/dubbo/org.apache.dubbo.rpc.Filter。

链路长这样：Proxy → ClusterFilter → ClusterInvoker → Filter → Invoker。这正是 Dubbo3 相对 2.x 的变化之一。

呼应 Part B ③

Filter 链就是 Wrapper 包装的运行时体现：你注册的多个 Filter，被像套娃一样一层层包在真实 Invoker 外面——createExtension 里那段 for (wc : cachedWrapperClasses) instance = new wc(instance); 的循环，套出来的就是这种链。
