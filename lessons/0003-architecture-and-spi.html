Part A：十层架构——源码的地图

Dubbo 是个大框架（几十个模块）。它靠分层来组织复杂度：每一层只做一件事，上层调用下层，单向依赖、互不穿透。记住一句话——「分而治之，向下依赖」。

业务层 · Business

  • Service 服务接口层：开发者写的业务接口与实现（你的 DemoServiceImpl）

RPC 层 · Dubbo 的核心（6 层）

  • Config 配置层：解析 ServiceConfig / ReferenceConfig
  • Proxy 代理层：生成 Consumer 的远程代理、Provider 的 Invoker
  • Registry 注册层：服务注册与发现（你用的 Nacos 就在这层）
  • Cluster 集群层：多 Provider 的路由 / 负载均衡 / 容错
  • Monitor 监控层：调用次数与耗时统计
  • Protocol 协议层：封装「如何暴露、如何调用」（Dubbo 协议 / Triple）

Remoting 层 · 远程通信（3 层）

  • Exchange 信息交换层：封装「请求-响应」语义
  • Transport 网络传输层：抽象 Netty 等 NIO 框架
  • Serialize 序列化层：Hessian2 / Fastjson2 / Protobuf

Part B：SPI 扩展机制——可插拔的钥匙

看上面的表你会发现：Protocol、Registry、LoadBalance、Serialization……几乎每一层的关键接口都是「可换」的。Dubbo 靠什么做到「换协议只改一行配置」？答案就是 SPI。

SPI（Service Provider Interface）一句话：把「接口」和「实现」解耦，实现类的全限定名写在配置文件里，运行时按需加载。

为什么不直接用 Java 自带的 SPI？

维度Java SPI（ServiceLoader）Dubbo SPI（ExtensionLoader）
加载一次性全量实例化所有实现按需、按 key 加载
取实现只能遍历，无法按名取getExtension("random") 按名直取
IOC 注入无有（setter 注入其他扩展）
AOP 包装无有（Wrapper 装饰，如 Filter 链）
自适应无有（@Adaptive，按 URL 参数运行时选）

正因 Java SPI「全量加载 + 不能按名取 + 无注入/包装/自适应」，Dubbo 才自己造了一套更强的。

Dubbo SPI 的约定（背下来）

  1. 扩展接口上标 @SPI（如 Protocol、LoadBalance）。
  2. 实现配置在 META-INF/dubbo/<接口全限定名> 文件里，格式 key=实现类全限定名。
  3. 每个 @SPI 接口对应一个 ExtensionLoader 实例，负责加载/缓存/注入。

看个真实的——Dubbo 源码里 LoadBalance 的 SPI 配置文件（你能在 dubbo-cluster jar 里找到它）：

# 文件：META-INF/dubbo/org.apache.dubbo.rpc.cluster.LoadBalance
random=org.apache.dubbo.rpc.cluster.loadbalance.RandomLoadBalance
roundrobin=org.apache.dubbo.rpc.cluster.loadbalance.RoundRobinLoadBalance
leastactive=org.apache.dubbo.rpc.cluster.loadbalance.LeastActiveLoadBalance
consistenthash=org.apache.dubbo.rpc.cluster.loadbalance.ConsistentHashLoadBalance
shortestresponse=org.apache.dubbo.rpc.cluster.loadbalance.ShortestResponseLoadBalance
这就是「换负载均衡只改一个 key」

当你配置 dubbo.consumer.loadbalance=random，Dubbo 就调用 ExtensionLoader.getExtensionLoader(LoadBalance.class).getExtension("random")，拿到 RandomLoadBalance。换策略 = 换一个 key，框架代码一行都不用动。换协议（dubbo.protocol.name）、换注册中心（registry 的 scheme）同理——全是在选 SPI 扩展点。

三个增强，先有印象

Dubbo SPI 还有三招，下节课细讲源码：① IOC——扩展的 setter 会被自动注入其它扩展；② AOP/Wrapper——用包装类包一层（Filter 链就是这么来的）；③ @Adaptive 自适应——一个「根据 URL 参数临时决定用哪个实现」的动态扩展（这是理解 Dubbo「面向接口 + URL 总线」的关键）。
