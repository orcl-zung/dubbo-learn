第 2 课 · 地基 / 阶段一

动手跑通第一个 Dubbo 应用

第 1 课是地图，这节课是亲自上路。你要亲手写一个 Provider、一个 Consumer，让它们隔着 JVM 完成一次调用——有了这次体感，后面读源码才不是读天书。

⏱ 约 30–45 分钟 使命 ← 第 1 课 架构速查
本课的 tangible win

启动两个进程，Consumer 终端打印出：

远程调用结果: Hello, dubbo

这一行，就是一次货真价实的跨进程远程方法调用——你完成了。

为什么是三个模块

Dubbo 是「面向接口」的 RPC：Consumer 调用的、Provider 实现的，必须是同一个接口（全限定类名完全一致）。所以接口要抽成一个共享的 jar，两端都依赖它。于是工程长这样：

dubbo-learn-practice/
├── dubbo-demo-api/         # 共享契约：DemoService 接口
├── dubbo-demo-provider/    # 提供者：实现 + 暴露
└── dubbo-demo-consumer/    # 消费者：注入代理 + 调用
前置条件

能跑 Java 的开发环境：具体是 JDK 17 或 21（LTS）+ Maven。本课 pom 里 <java.version>17</java.version> 只是「编译目标」，在你的 JDK 21 上直接运行，无需任何改动（低版本字节码在高版本 JVM 上照跑）。还在 JDK 8？把 spring-boot-starter-parent 换成 2.7.x，Dubbo 版本不变，代码完全一样。

先看全貌：依赖关系 & 正确的建立顺序

在贴代码前，先把三个模块的关系和怎么建讲清楚，后面才不会乱。

dubbo-demo-api 是什么？——「契约 jar」

Dubbo 是面向接口的 RPC：Consumer 持有的代理、Provider 实现的，必须是同一个接口（全限定类名完全一致）。所以这个接口要同时出现在两个进程的 classpath 上。最干净的做法——把它抽成一个独立 jar（dubbo-demo-api），里面只有接口、没有实现、不依赖任何东西，两头各自依赖它。这就是微服务里标准的「API 契约模块」。

      dubbo-demo-provider          dubbo-demo-consumer
              | depends on                | depends on
              |                           |
              +-------------+-------------+
                            |
                     dubbo-demo-api      (只有接口；不依赖任何东西)

provider 与 consumer 互不依赖——它们是两个独立进程，只靠「同一个接口 + 同一个注册中心」耦合；各自还依赖 dubbo starter 与注册中心客户端。

你说得对：建项目本就该 pom 先行

Maven 项目先写 pom（坐标 + parent + 依赖），再写代码。下面我把每个模块的 pom 和代码贴在一起，只是方便你对照——真正的建立顺序是 pom-first，而且有个不能跳的先后：

1) dubbo-demo-api      : pom (坐标 + Java 版本) -> 写 DemoService 接口
2) mvn install         : 把 api 装进本地 Maven 仓库      (关键! 别跳)
3) dubbo-demo-provider : pom 声明依赖 dubbo-demo-api (此刻才解析得到) -> 写实现
4) dubbo-demo-consumer : 同样 pom 先 -> 写代码

为什么必须先 mvn install api？因为 provider/consumer 的 pom 写了「依赖 com.example:dubbo-demo-api:1.0-SNAPSHOT」，Maven 得在本地仓库里找到这个 jar 才能编译。api 没装进去，provider 一编译就报「找不到 DemoService」——这正是「第 3 步：跑起来」第一行先 cd dubbo-demo-api && mvn install 的原因。

第 0 步：共享契约 dubbo-demo-api

一个普通 Java 接口，没有任何依赖。

package com.example;

public interface DemoService {
    String sayHello(String name);
}

dubbo-demo-api/pom.xml（极简，能编译成 jar 即可）：

<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0">
    <modelVersion>4.0.0</modelVersion>
    <groupId>com.example</groupId>
    <artifactId>dubbo-demo-api</artifactId>
    <version>1.0-SNAPSHOT</version>
    <properties>
        <maven.compiler.source>17</maven.compiler.source>
        <maven.compiler.target>17</maven.compiler.target>
    </properties>
</project>

第 1 步：提供者 dubbo-demo-provider

实现接口，用 @DubboService 把它暴露成 Dubbo 服务。

package com.example.provider;

import com.example.DemoService;
import org.apache.dubbo.config.annotation.DubboService;

@DubboService            // 关键：触发「服务暴露」并注册到注册中心
public class DemoServiceImpl implements DemoService {
    @Override
    public String sayHello(String name) {
        return "Hello, " + name;
    }
}
package com.example.provider;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class ProviderApplication {
    public static void main(String[] args) {
        SpringApplication.run(ProviderApplication.class, args);
    }
}

dubbo-demo-provider/src/main/resources/application.yml：

dubbo:
  scan:
    base-packages: com.example   # 扫描 @DubboService，触发服务暴露（关键！不加就不暴露）
  application:
    name: demo-provider
  protocol:
    name: dubbo          # 用 Dubbo2 协议（基于 TCP）
    port: 20880
  registry:
    address: zookeeper://127.0.0.1:2181   # 注册中心
头号坑：必须开扫描，@DubboService 才会暴露

Dubbo 默认不会自动扫描 @DubboService——必须配 dubbo.scan.base-packages（或在主类加 @EnableDubbo），Dubbo 才会把实现类注册成 ServiceBean 并暴露。否则 Provider 看着在跑、日志里却没有 Export dubbo service、Nacos 里也找不到，Consumer 报 No provider。本项目调试时亲历此坑。

两个端口别搞混

这个 Provider 其实开了两个监听端口，分属两套不同的服务：

  • dubbo.protocol.port = 20880：Dubbo 自己的 Netty RPC 端口，真正的 RPC 流量走这里，Consumer 直连的就是它（注册到注册中心的也是这个端口）。
  • server.port = 8080（默认，未显式写）：Spring Boot 的内嵌 Tomcat（HTTP）端口，给 @RestController 用。本课没写任何 HTTP 接口，它纯粹是为了让进程常驻而存在。

一句话：Consumer 连的是 20880，跟 server.port 的 Tomcat 毫无关系——这正是「Dubbo 有自己的通信栈，独立于 HTTP/Tomcat」。想更干净可加 spring.main.web-application-type: none 关掉 Tomcat。

dubbo-demo-provider/pom.xml：

<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0">
    <modelVersion>4.0.0</modelVersion>
    <parent>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-parent</artifactId>
        <version>3.2.4</version>
    </parent>
    <groupId>com.example</groupId>
    <artifactId>dubbo-demo-provider</artifactId>
    <version>1.0-SNAPSHOT</version>
    <properties>
        <java.version>17</java.version>
        <dubbo.version>3.3.4</dubbo.version>
    </properties>
    <dependencies>
        <!-- 用 web starter 让 Provider 进程常驻（Tomcat 兜底）-->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
        </dependency>
        <dependency>
            <groupId>org.apache.dubbo</groupId>
            <artifactId>dubbo-spring-boot-starter</artifactId>
            <version>${dubbo.version}</version>
        </dependency>
        <dependency>
            <groupId>org.apache.dubbo</groupId>
            <artifactId>dubbo-registry-zookeeper</artifactId>
            <version>${dubbo.version}</version>
        </dependency>
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>dubbo-demo-api</artifactId>
            <version>1.0-SNAPSHOT</version>
        </dependency>
    </dependencies>
</project>
关于 dubbo.version

<dubbo.version> 是我们自己定义的 Maven 属性（写在 <properties> 里），不是框架自动给的——${dubbo.version} 构建时被替换成这个值。取值去 Maven Central / MVNRepository 查最新 3.3.x（写本课时是 3.3.4，你换更新的即可；也建议对齐你将来读源码的 tag）。所有 Dubbo 模块必须同一个版本，所以抽成属性一处管控。注意：Spring Boot 的 starter 由它自己的 BOM 托管、可以不写版本；但 Dubbo 不在 Spring Boot 的 BOM 里，必须显式写。（进阶：在 <dependencyManagement> 导入官方 dubbo-bom 后，Dubbo 模块即可省略版本。）

第 2 步：消费者 dubbo-demo-consumer

用 @DubboReference 注入一个远程服务的本地代理，然后像调本地方法一样调它。

package com.example.consumer;

import com.example.DemoService;
import org.apache.dubbo.config.annotation.DubboReference;
import org.springframework.boot.CommandLineRunner;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class ConsumerApplication implements CommandLineRunner {

    @DubboReference          // 关键：注入远程服务的「代理」，背后是网络
    private DemoService demoService;

    public static void main(String[] args) {
        SpringApplication.run(ConsumerApplication.class, args);
    }

    @Override
    public void run(String... args) {
        String result = demoService.sayHello("dubbo");  // 像调本地方法一样调远程
        System.out.println("远程调用结果: " + result);
    }
}

dubbo-demo-consumer/src/main/resources/application.yml：

dubbo:
  application:
    name: demo-consumer
  registry:
    address: zookeeper://127.0.0.1:2181   # 订阅同一个注册中心
  consumer:
    timeout: 3000
Consumer 的 pom

与 Provider 几乎一样，只把 spring-boot-starter-web 换成 spring-boot-starter（让 Consumer 跑完打印就退出，干净利落）。其余依赖（dubbo starter、registry-zookeeper、api）完全相同。

第 3 步：跑起来

# 0) 启动 Zookeeper（装了 Docker 就一条命令；没装见下方说明）
docker run -d --name zk -p 2181:2181 zookeeper

# 1) 把共享 api 装进本地仓库（Provider/Consumer 都依赖它）
cd dubbo-demo-api && mvn clean install

# 2) 启动 Provider —— 终端 1，保持运行，看到日志稳定后别关
cd ../dubbo-demo-provider && mvn spring-boot:run

# 3) 启动 Consumer —— 另开终端 2
cd ../dubbo-demo-consumer && mvn spring-boot:run

Consumer 终端会打印出：

远程调用结果: Hello, dubbo
头号常见报错：No provider available

Consumer 启动时报「找不到某服务的 Provider」= 此刻注册中心里没有该接口的提供者。按概率排查：①启动顺序——Provider 先起、等到日志出现 Export dubbo service ... 再起 Consumer；②Provider 日志确实注册成功且 Zookeeper 在跑；③接口全限定名两边一致（共享 api jar 自动一致）；④同一个 registry.address。临时绕过可设 dubbo.consumer.check: false，但根因仍要修。

没有 Docker？

从 Zookeeper 官网下二进制包，解压后 cp conf/zoo_sample.cfg conf/zoo.cfg && ./bin/zkServer.sh start。或换成 Nacos：sh startup.sh -m standalone，再把两处 registry.address 改成 nacos://127.0.0.1:8848、依赖换成 dubbo-registry-nacos。

把这次运行对回第 1 课

对照上面的图，你刚刚亲手走完了真实链路：

  • Provider 启动 → 步骤 0+1：暴露服务（监听 20880），并把地址注册进 Zookeeper。
  • Consumer 启动 → 步骤 2+3：向 Zookeeper 订阅，收到 Provider 的地址推送。
  • demoService.sayHello("dubbo") → 步骤 4：Consumer 直连 Provider发起 RPC 调用。
  • Hello, dubbo ← 这就是远程方法跨 JVM 返回的结果。
关键验证

第 1 课那句铁律——「注册中心只管地址，调用是 Consumer 直连 Provider」——你这次用真实代码验证了：Zookeeper 只是帮忙递了张地址条，真正的数据流量没经过它。

进阶观察（可选）

想看 Zookeeper 里到底注册了什么？用 Dubbo Admin 控制台最直观；或 docker exec -it zk zkCli.sh 进 ZK 命令行翻节点。注意：Dubbo3 默认是应用级服务发现，节点结构与传统 2.x 的接口级不同——这正是后面「服务发现」专题要深挖的，现在先有个印象即可。
