RPC 与 Dubbo：它解决了什么问题？

先别碰源码。这节课把「Dubbo 到底在干嘛」变成一句话、一张图——这是后面所有源码阅读的地基。

⏱ 约 10 分钟 使命 术语表 架构速查

从一个本地方法调用说起

你写过无数次这样的代码：

// 同一个 JVM 里， userService 是个普通对象
User user = userService.getById(1L);

现在换个场景：UserService 的实现不在你这台机器上，而在另一台机器的另一个 JVM 里。你希望调用它的代码看起来几乎一样：

@DubboReference          // 注入的是一个「代理」，背后是网络
private UserService userService;

User user = userService.getById(1L);  // 像调本地方法一样调远程
核心承诺

RPC（Remote Procedure Call，远程过程调用）的承诺：让你像调用本地方法一样调用远程方法。 它把「序列化参数 → 网络传输 → 对端执行 → 反序列化结果」这一长串脏活全藏起来。Dubbo 就是这样一个 RPC 框架。

那为什么不直接用 HTTP / Socket？

因为只有「能调通」是不够的。生产环境里 Provider 不止一台、会上下线、会挂掉，你需要解决一堆新问题：

  • 服务发现：Consumer 怎么知道有哪些 Provider、它们在哪些 IP 上？Provider 上下线怎么实时感知？
  • 负载均衡：多台 Provider，这次请求该打到哪一台？
  • 集群容错：某一台 Provider 挂了/超时了，怎么办？自动换一台重试？还是快速失败？
  • 可观测 / 治理：调用量、耗时、灰度路由、限流降级怎么管？
一句话定义 Dubbo

Dubbo = RPC（把远程调用做成像本地调用） + 一整套服务治理能力（发现、负载均衡、容错、治理、可观测）。这些治理能力正是你将来读源码时要逐个攻克的模块。

五个角色与一次调用

Dubbo 经典架构里有五个角色，一次服务从启动到被调用，时序如下：

角色

  • Container 服务容器：启动、加载、运行 Provider（Spring 体系下就是 Spring 容器）。
  • Provider 服务提供者：暴露服务、监听端口、执行业务。
  • Registry 注册中心：协调地址注册与发现（如 Zookeeper / Nacos）。
  • Consumer 服务消费者：发起调用，持有远程服务的本地代理。
  • Monitor 监控中心：统计调用次数与耗时。

时序（0 → 5）

  1. Container 启动 Provider。
  2. Provider 把服务地址（URL）注册到 Registry。
  3. Consumer 向 Registry 订阅所需服务。
  4. Registry 通知 Consumer：把地址列表推过去，变更时实时推送。
  5. Consumer 按负载均衡选一台 Provider 直连调用；失败按容错策略重试。
  6. Consumer 与 Provider 每分钟把统计上报到 Monitor。
最重要的一句话

注册中心只管「地址」，真正的调用是 Consumer 直连 Provider，流量不经过注册中心。 记死这条——它是后面理解「为什么注册中心挂了已发现的地址还能用」、以及读 Cluster/LoadBalance 源码的前提。
