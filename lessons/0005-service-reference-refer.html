第 5 课 · 阶段二 / 站 B

服务引用 refer:这条 Invoker 链是谁搭的

第 4 课你追了「一次调用怎么走」(inside-out);这节课补另一半——这条 Invoker 链是谁、在哪一步、按什么顺序搭起来的(outside-in)。答案就在 @DubboReference 背后的 ReferenceConfig#get。读完,consumer 侧闭环。

⏱ 约 25–35 分钟 ← 第 4 课 调用全链路(inside-out) 攻读蓝图
本课的 tangible win

能徒手画出两幅图并说清它们的关系:

  • 装配链(refer 期,outside-in):DubboInvoker → 实例 filter → RegistryDirectory → cluster.join → ClusterFilter → MockClusterInvoker → getProxy
  • 调用链(第 4 课,inside-out):proxy → InvokerInvocationHandler → MockClusterInvoker → … → DubboInvoker → 网络

两条链是同一根链的正反两面——这是本课的「啊哈」。

先定位:refer 是「搭链子」,不是「走链子」

一次 Dubbo 调用分两段:装配期(refer,@DubboReference 时把链搭好)和调用期(你调 sayHello 时在链上走)。第 4 课讲调用期,本课讲装配期。下节课的 export 是 refer 在 provider 侧的镜像。别把「搭」和「走」混了。

Part A · 入口:从 @DubboReference 到 ReferenceConfig#get

你写的 @DubboReference DemoService demoService,Spring 启动时由 ReferenceAnnotationBeanPostProcessor 处理,为它建一个 ReferenceConfig(或从 ReferenceConfigCache 取)。真正的装配是懒加载的——ReferenceConfig#get 第一次被调用(ref == null)时才发生:

// ReferenceConfig.java
public T get(boolean check) {
    if (destroyed) throw ...;
    if (ref == null) {
        ...getScopeModel().getDeployer().start();   // 确保模块启动
        init(check);                                 // ★ 首次 get 才真正装配
    }
    return ref;
}

protected void init(boolean check) {
    ...refresh 配置、建 ConsumerModel 并注册...
    ref = createProxy(referenceParameters);          // ★ 装配主干
    ...
    if (check) checkInvokerAvailable(0);             // check=true 时校验有无 provider
    initialized = true;
}
所以「什么时候搭链子」

不是 Spring 启动那一瞬,而是 ReferenceConfig#get 首次调用时——通常就是代理注入完成、你第一次(或启动校验时)碰到它的那一刻。check 默认 true,所以装配完会立刻校验「有没有 provider」,没有就抛 No provider available(第 2 课那个头号报错的源头之一)。

Part B · 装配主干:createProxy → createInvoker → refer

init 调 createProxy,它做两件事:先把 invoker 建出来,再用 invoker 建代理:

private T createProxy(Map<String,String> referenceParameters) {
    ...
    if (StringUtils.isNotEmpty(url)) parseUrl(...);        // 点对点直连 url
    else aggregateUrlFromRegistry(...);                     // 从注册中心拼 URL(常态)
    createInvoker();                                        // ★ 建 this.invoker
    ...
    // ★ 用 invoker 造代理
    return (T) proxyFactory.getProxy(invoker, ProtocolUtils.isGeneric(generic));
}

createInvoker 按 URL 类型分两条路(这是理解装配的关键分叉):

private void createInvoker() {
    if (urls.size() == 1) {
        URL curUrl = urls.get(0);
        // ★ registry 场景:refer 返回的【已经是集群包装好的 invoker】
        invoker = protocolSPI.refer(interfaceClass, curUrl);
        if (!UrlUtils.isRegistry(curUrl)
                && !curUrl.getParameter(UNLOAD_CLUSTER_RELATED, false)) {
            // 只有【直连 URL】才需要这里再包一层 cluster + StaticDirectory
            invoker = Cluster.getCluster(getScopeModel(), Cluster.DEFAULT)
                    .join(new StaticDirectory(curUrl, List.of(invoker)), true);
        }
    } else {
        // 多注册中心:每个 refer,再用 ZoneAwareCluster.join(StaticDirectory([invokers])) 汇总
        ...
    }
}
读出这层的潜台词

registry 场景下 createInvoker 没有再调 cluster.join(注意 !isRegistry 守卫)——说明 protocolSPI.refer(registryUrl) 返回的那个 invoker,自身就已经是集群包装好的。也就是说,cluster.join(directory) 发生在 refer 内部。下一节我们就钻进去看。

Part C · 钻进 RegistryProtocol.refer:directory + cluster.join

protocolSPI 是自适应的 Protocol。对 registry:// URL,它派发到 RegistryProtocol(继承 InterfaceCompatibleRegistryProtocol)。RegistryProtocol.refer(doRefer)做三件事——这是站 B 的心脏:

RegistryProtocol.doRefer(interface, registryUrl)
   │
   ├─① new RegistryDirectory(interface, registryUrl)        一个「动态目录」,持有当前所有 provider 的 Invoker
   │
   ├─② directory.subscribe(url)
   │       └─ 注册中心 notify(provider URLs) → directory 刷新 Invoker 列表:
   │            每个 provider URL → 真正的 protocol.refer  →  DubboInvoker(发起网络的那个)
   │            每个DubboInvoker 被 ProtocolFilterWrapper 包上【实例级 filter 链】(你的 TimingFilter 挂这)
   │
   └─③ cluster.join(directory)
           └─ 把 directory 包成 ClusterInvoker:
                ClusterFilter 链(pre-LB)+ FailoverClusterInvoker(directory)
                外层再套 MockClusterWrapper → MockClusterInvoker
   │
   └─ return 这个【集群 invoker】给 createInvoker

这里每一步你在前几课都见过,现在它们被串成了一条装配流水线:

  • DubboInvoker:每个 provider 地址一个,真正发网络(细节留到站 F)。
  • 实例 filter 链:由 ProtocolFilterWrapper.refer 用 buildInvokerChain(..., REFERENCE_FILTER_KEY, CONSUMER) 包在每个 DubboInvoker 外——你第 4 课 Part D 写的 TimingFilter(@Activate(group=CONSUMER) 的 Filter)就注册在这。
  • cluster.join:第 4 课读过——AbstractCluster.join = buildClusterInterceptors(doJoin(directory)),doJoin 出 FailoverClusterInvoker,外面叠 ClusterFilter 链(pre-LB,ClusterFilterInvoker)。
  • MockClusterWrapper:Cluster SPI 的 Wrapper,再套一层 MockClusterInvoker 处理 mock——这正是第 4 课调用链最外层那个 MockClusterInvoker 的来源。

Part D · 收口:proxyFactory.getProxy

回到 createProxy 最后一行:proxyFactory.getProxy(invoker)(默认 Javassist)生成代理,handler = new InvokerInvocationHandler(invoker)。

闭环到第 4 课

第 4 课你读过的 InvokerInvocationHandler,它那个 private final Invoker<?> invoker 字段——装的就是这一节 createInvoker 搭出来的那整串集群 invoker(MockClusterInvoker → ClusterFilter → FailoverClusterInvoker → directory → 实例filter → DubboInvoker)。装配端塞进去什么,调用端就从什么开始走。两节课在此刻接上。

至此 ref = 代理,注入到你的 @DubboReference 字段。装配结束。

Part E · 啊哈:装配 outside-in ↔ 调用 inside-out

把本课(装配)和第 4 课(调用)叠在一起看,会看到一根链的正反两面:

【装配期 · refer · 从内往外搭(outside-in)】
   DubboInvoker  ──包──►  实例 filter 链
        └──装进──►  RegistryDirectory(动态,provider 变就刷新)
                          └──cluster.join──►  ClusterFilter 链 + FailoverClusterInvoker
                                                  └──MockClusterWrapper──►  MockClusterInvoker
                                                                          └──getProxy──►  代理(InvokerInvocationHandler)

【调用期 · sayHello · 从外往里走(inside-out,第 4 课)】
   代理  ──►  InvokerInvocationHandler  ──►  MockClusterInvoker  ──►  ClusterFilter
            ──►  FailoverClusterInvoker(doSelect 选 LB)  ──►  directory.list
            ──►  实例 filter 链  ──►  DubboInvoker  ──►  网络
这就是 consumer 侧的完整闭环

装配期从内向外把 Invoker 一层层包好(先有 DubboInvoker,再有 directory,再 cluster.join,再 getProxy);调用期从外向内一层层调用进去。方向相反,链是同一条。看懂这张对照图,consumer 侧就彻底通了——从「这条链是谁搭的」到「一次调用怎么走」,首尾呼应。

Part F · 点到为止:3.x 的两个「双」(深化留后面)

双服务发现模型(站 C 详讲)

3.x 同时支持两套服务发现:

  • 接口级(2.x 兼容):RegistryProtocol + RegistryDirectory,按接口订阅 provider URL。
  • 应用级(3.x 新):ServiceDiscoveryRegistryDirectory,消费应用级 ServiceInstance 数据,经 InstanceAddressURL 转成 Invoker。URL 用 service-discovery-registry 协议时走这条。

本课为简洁以接口级为主线;应用级、以及两者如何迁移兼容(精妙处),留到站 C。
