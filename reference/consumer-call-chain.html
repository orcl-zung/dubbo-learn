Consumer 调用全链路:从 proxy 到一次 RPC

架构速查是「静态分层地图」,这页是「动态一次调用图鉴」——把 demoService.sayHello() 从代理对象到网络包的每一步钉死,辨清 Proxy / InvokerInvocationHandler / Invoker 三种角色,并说清它和 JDK 动态代理的关系。版本以 Dubbo 3.3.x 源码为准。

一句话总记

一次远程调用 = 1 个 InvocationHandler(在代理边界把 Method 翻译成 RpcInvocation) + 一条 Invoker 套娃链(每层 invoke(Invocation) 转发内层)。门面是 InvokerInvocationHandler,门面之后全是 Invoker。

一图总览:一次调用的完整链路

demoService.sayHello("lea")              你调用的「业务接口」对象(其实是代理)
      │   JDK/Javassist 代理把方法调用转发给它内部的 handler
      ▼
InvokerInvocationHandler.invoke(proxy, method, args)        <── 全链唯一的 InvocationHandler
      │   ① Object 方法(toString/hashCode/equals)在此短路,不下发
      │   ② 否则:new RpcInvocation(method, args)
      │      InvocationUtil.invoke(invoker, rpcInvocation)   invoker = Invoker 链的头
      ▼ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ 代理世界 → Invoker 世界的边界
      ▼
MockClusterInvoker.invoke(Invocation)                       <── Invoker 链开始(全是 Invoker)
      │   无 mock 配置 → this.invoker.invoke(invocation)     透传给内层
      ▼
[Consumer Filter 链: ConsumerContextFilter / MonitorFilter / FutureFilter / 你的自定义 Filter…]
      │   每个 Filter 被包成一个 Invoker 节点,链式转发
      ▼
AbstractClusterInvoker.invoke(Invocation)                    FailoverClusterInvoker 继承它
      │   ① list(invocation)      Directory 给出「路由后」的 provider Invoker 列表
      │   ② initLoadBalance(...)  ★ getExtension("first") → 首次 new FirstLoadBalance(实例化日志)
      │   ③ doInvoke(invocation, invokers, loadbalance)
      ▼
FailoverClusterInvoker.doInvoke(...)                         重试循环(retries,默认 2 次)
      │   select(loadbalance, invocation, copyInvokers, invoked)
      ▼
AbstractClusterInvoker.doSelect(...)
      │   invokers.size()==1 ? → return invokers.get(0)      【first.select 不调】
      │   ≥2                  → loadbalance.select(...)      ★ FirstLoadBalance.select(select 日志)
      ▼
选中的 provider Invoker → invokeWithContext → Protocol → Remoting(Netty) → Provider

消费端这条链是「装饰器套娃」:每一层 .invoke 都把请求往内层传,直到真正的集群逻辑与网络发送。

三种角色:Proxy / InvokerInvocationHandler / Invoker(链)

最容易混淆的就是这三个。记住:代理 不是 Invoker——代理实现的是业务接口,Invoker 是装在它肚子里那个 handler 里的。

角色implements干什么数量
代理对象 demoService业务接口(DemoService)+ EchoService/Destroyable用户看到的「本地方法」门面;让远程调用长得像本地调用每个 @DubboReference 1 个
InvokerInvocationHandlerjava.lang.reflect.InvocationHandler边界适配器:Method→RpcInvocation;Object 方法短路;持有 1 个 Invoker每个代理 1 个(全链唯一那种)
Invoker 链各节点Dubbo Invoker<T>invoke(Invocation) 转发内层;承载 filter / cluster / loadbalance一条链
is-a / has-a 记法

代理 is-a 业务接口;InvokerInvocationHandler has-a Invoker;Invoker 链每一层 is-a Invoker。三者是不同类型,InvokerInvocationHandler 是把它们粘起来的适配器。

这套设计背后的哲学:「一切皆 Invoker,Invocation 在一条 Invoker 链上流动」。代理负责「伪装成业务接口」(强类型、像调本地),Invoker 链负责「统一管线」(可透明叠 filter/cluster/lb),handler 负责在两套形状之间翻译一次。

SPI 扩展在链路里何时被加载/使用(以 first LoadBalance 为例)

first 的「加载」和「使用」是链路上两个不同时刻,而且都发生在集群层内部——不经过 InvokerInvocationHandler。

位置触发条件
加载(实例化)AbstractClusterInvoker.invoke 的 initLoadBalance每次调用前都跑;首次触发 getExtension("first") → new FirstLoadBalance(你的实例化日志)。之后走缓存
使用(select)doInvoke → select → doSelect受 size()==1 短路影响:单 provider 不调 select;≥2 个才调

getExtension("first") 命中缓存就直接返回;首次访问走 createExtension,三步:

private T createExtension(String name, boolean wrap) {
    Class<?> clazz = getExtensionClasses().get(name);     // ① 按名找类(首次触发读 SPI 配置)
    T instance = extensionInstances.get(clazz);           //    反射 new(单例,按 Class 缓存)
    if (instance == null) { ...; injectExtension(instance); }  // ② IOC:setter 注入别的扩展
    if (wrap)
        for (Class<?> wc : cachedWrapperClasses)         // ③ AOP:Wrapper 逐层包(Filter 链的由来)
            instance = wc.getConstructor(type).newInstance(instance);
    return instance;
}
单 provider 短路(高频踩坑)

AbstractClusterInvoker.doSelect 对 invokers.size()==1 直接 return invokers.get(0),不调 loadbalance.select。所以单 provider 时你只看到「实例化日志」、看不到「select 日志」——但 initLoadBalance 每次仍按 key 加载实例,足以证明扩展被选中。要看 select 真跑,起第 2 个 provider(覆盖 dubbo.protocol.port)。

跟 JDK 动态代理的关系:复用 + 换芯(不是「重写」)

严格说 Dubbo 没有「重写」JDK 动态代理,而是复用其模式、可替换其生成器:

  • jdk 工厂:Proxy.newProxyInstance(loader, interfaces, new InvokerInvocationHandler(invoker)) —— 直接用 JDK 的 Proxy,handler 换成 Dubbo 的。
  • javassist 工厂(默认):Proxy.getProxy(interfaces).newInstance(new InvokerInvocationHandler(invoker)) —— 代理类用 Javassist 字节码生成,但派发照样落到同一个 InvokerInvocationHandler。

选哪个由 ProxyFactory 这个 SPI 决定(@SPI("javassist") + @Adaptive(proxy))——「换代理实现 = 换 proxy 这个 SPI key」,和换 LoadBalance 是同一机制。

你手写的 JDK 动态代理Dubbo consumer 代理
代理怎么生成Proxy.newProxyInstanceProxyFactory.getProxy(jdk 同 JDK;javassist 字节码)
handler你的 lambda(InvocationHandler)InvokerInvocationHandler(也是 InvocationHandler)
handler 里「真正干活」method.invoke(new HelloImpl(), args) 本地反射invoker.invoke(new RpcInvocation(...)) → Invoker 链 → 网络 RPC
Object 方法不处理(toString 也走 handler)短路(不发 RPC)
持有的「真实目标」new HelloImpl()(本地实现)Invoker 链(通往远端 Provider)
点睛

Dubbo 的 consumer 代理 = 你的 JDK 动态代理,只是把 handler 里那行 method.invoke(真实对象) 从「本地反射调真实对象」换成「交给 Invoker 链发一次 RPC」。代理骨架(Proxy + InvocationHandler)完全是 JDK 那套。

自验证断点清单(按调用顺序)

  1. InvokerInvocationHandler#invoke —— 看它把 Method 转 RpcInvocation 后调 invoker.invoke
  2. MockClusterInvoker#invoke —— 看 this.invoker.invoke 透传
  3. AbstractClusterInvoker#invoke —— 看 initLoadBalance 取出 first、再 doInvoke
  4. FailoverClusterInvoker#doInvoke —— 看重试循环里的 select(...)
  5. AbstractClusterInvoker#doSelect —— 看 size()==1 短路 / loadbalance.select

真正用上 first 的是 step 3(加载) 和 step 5(select);step 1/2 只是「把请求往里传」。
