角色与部署

Provider服务提供者

暴露并对外提供 RPC 服务的 Dubbo 进程，监听端口、接收调用、执行业务逻辑。

Consumer服务消费者

发起 RPC 调用的 Dubbo 进程。持有远程服务的本地代理，像调本地方法一样调远程。

Registry注册中心

协调服务地址注册与发现的中心组件。Provider 注册地址，Consumer 订阅地址，变更时推送。常用 Zookeeper / Nacos。

Monitor监控中心

统计 Consumer 与 Provider 的调用次数、耗时等指标，每分钟上报一次，用于可观测。

Container服务容器

负责启动、加载、运行 Provider。在 Spring 体系里通常就是 Spring 容器。

三大中心

Dubbo3 传统部署中的中心化组件：注册中心（地址发现）、配置中心（全局配置 + 治理规则）、元数据中心（接口元数据）。三者都不是必须的，但通常至少有一个注册中心。

核心动作

服务暴露 ExportProvider 端动作

Provider 启动时，把服务以本地（Injvm）和远程两种方式暴露成 Invoker，并注册到注册中心。源码入口：ServiceConfig#export。

服务引用 ReferConsumer 端动作

Consumer 启动时，从注册中心订阅地址，为远程接口生成代理对象注入业务代码。源码入口：ReferenceConfig#get。

Invoker可执行体

Dubbo 的核心抽象：一个「可被调用的实体」。物理Invoker 跑网络，Mock/Local Invoker 不跑。调用链就是层层 Invoker 包装。

服务发现 Service Discovery地址发现

Consumer 自动获取 Provider 地址列表的能力。Dubbo3 采用应用级模型（应用→实例列表），区别于 Dubbo2 的接口级（接口→实例列表）。

扩展与协议

SPI / 扩展点Service Provider Interface

Dubbo 的灵魂机制：把协议、注册中心、负载均衡等全部做成可插拔扩展点。比 Java SPI 更强（按名加载、按需实例化、自适应）。源码核心：ExtensionLoader。

自适应扩展 Adaptive运行时决策

一种特殊扩展：在调用时根据 URL 参数动态决定用哪个实现。是理解 Dubbo「面向接口 + URL 总线」设计的关键。

URL配置总线

Dubbo 内部贯穿一切的数据结构（dubbo://...?key=value）。扩展点的选择、参数传递都依赖它。URL 是 Dubbo 的血液。

Protocol协议

定义「如何暴露服务、如何引用服务、如何通信」。内置 Dubbo2（TCP 二进制）与 Triple（HTTP/2，兼容 gRPC）。

Triple 协议Dubbo3 主推

基于 HTTP/2 构建，完全兼容 gRPC，支持一元调用与三种流式通信。是 Dubbo3 云原生的基石。

Filter过滤器

调用链上的拦截器（类似 Servlet Filter）。监控、限流、日志、权限都做成 Filter，链式执行。

序列化 Serialization编解码

把方法参数/返回值转成字节流。Dubbo 不绑定序列化，支持 Hessian2、Fastjson2、Kryo、Protobuf 等。

集群与容错

Cluster集群

Consumer 侧的抽象：把多个 Provider Invoker 伪装成一个，并叠加容错策略。源码：Cluster / FailoverClusterInvoker。

集群容错策略Failover / Failfast ...

调用失败时怎么办。默认 Failover（失败自动切换重试）；还有 Failfast、Failsafe、Failback、Forking、Broadcast。

LoadBalance负载均衡

从多个 Provider 里选一个调用。内置 Random、RoundRobin、LeastActive、ConsistentHash、ShortestResponse 等。源码：LoadBalance。

Router路由

在负载均衡之前，先按规则（条件、标签、脚本）筛选出一组合法的 Provider，实现灰度、同区域优先等。

调用方式

泛化调用 Generic无接口调用

不依赖 Provider 的接口 SDK，用 GenericService 以方法名+参数发起调用。网关、测试平台常用。

直连点对点

不经过注册中心，Consumer 直接配置 Provider 的 IP:端口调用。调试或不需要服务发现时用。

为什么先记术语

这张表对应你源码之旅的每一站。当某节课深入某个机制时，回到这里确认它的定义与边界——避免把「服务暴露」和「服务注册」混为一谈这类常见错误。
