经典五角色与调用流程

来源：Apache Dubbo 官方工作原理图（重绘）。0–5 为一次完整服务的生命周期时序。

六步时序

  1. 启动：Container 启动、加载并运行 Provider。
  2. 注册：Provider 启动时把自己的服务地址（URL）注册到 Registry。
  3. 订阅：Consumer 启动时向 Registry 订阅所需服务。
  4. 通知：Registry 把 Provider 地址列表推给 Consumer，地址变更时基于长连接实时推送。
  5. 调用：Consumer 从地址列表中按负载均衡算法选一台 Provider 发起调用；失败则按容错策略重试。
  6. 统计：Consumer 与 Provider 在内存累计调用次数/耗时，每分钟上报一次到 Monitor。
关键认知

注册中心只管「地址」，调用是 Consumer 直连 Provider 的——流量不经过注册中心。所以注册中心挂了，已发现的地址仍能继续调用（只是无法感知新变更）。这条会被反复考到。

两个面（Dubbo3 抽象）

面是什么包含
数据面集群里所有跑业务的 Dubbo 进程Provider + Consumer（通过 RPC 协议交换数据）
治理控制面协调、管控这些进程的抽象层注册中心、配置中心、元数据中心、流量管控、Admin 控制台（Mesh 场景含 Istio）

三大中心职责

中心核心职责
注册中心地址注册与发现（Consumer↔Provider）。Client-Based 模式。
配置中心① 启动期全局配置（跨环境一致）② 治理规则（路由、动态配置）的存储与推送。
元数据中心① 接收 Provider 的接口元数据（供 Admin 服务测试/文档）② 作为应用级服务发现的补充。
默认行为

若不单独部署配置/元数据中心，Dubbo 会默认把注册中心实例同时当作配置中心与元数据中心（可用 use-as-config-center / use-as-metadata-center 关闭）。

同层对照：Dubbo vs Spring Cloud OpenFeign

真正公平的对比是 Dubbo ↔ OpenFeign（Spring Cloud 里负责「把远程调用封装成接口」的那块），而不是 Dubbo ↔ 整个 Spring Cloud 生态。编程模型上二者已趋同：都是「接口 + 动态代理，像调本地方法」。

维度DubboOpenFeign
是什么完整 RPC 框架（provider + consumer，通信+治理一体）仅 consumer 侧声明式 HTTP 客户端
传输协议TCP/二进制（Dubbo2）、HTTP/2（Triple）HTTP/REST（JSON 文本）
provider 侧一等公民：暴露服务、监听、注册没有，对端是普通 @RestController
负载均衡自带 LoadBalance SPI借 Spring Cloud LoadBalancer（原 Ribbon）
服务发现自带注册中心抽象借 Spring Cloud 注册中心组件
容错/熔断自带集群容错（Failover…）借 Resilience4j / Sentinel
性能高（长连接、二进制）低一截（HTTP 文本开销）
一句话本质

OpenFeign 是「拼装体系」里的一块零件（只管 接口→代理→HTTP，其余能力靠兄弟组件借）；Dubbo 是「端到端自研」（provider、consumer、传输、发现、负载、容错全内置、靠 SPI 串成一体）。差异收敛成三点：传输方式（RPC vs HTTP）、集成度（一体 vs 拼装）、provider 侧是否一等公民。

读源码的锚点

Consumer 侧二者结构对称——都是「接口 → 动态代理 → 调用处理器」：

@FeignClient 接口 → FeignClientFactoryBean → JDK 动态代理 → InvocationHandler → HTTP
@DubboReference    → ReferenceConfig          → 代理(ProxyFactory) → InvokerInvocationHandler → Invoker 链 → 通信

区别在代理背后：OpenFeign 直接拼 HTTP；Dubbo 走 Invoker 链 → Cluster → LoadBalance → Protocol → Remoting。记住这条对应，将来读 InvokerInvocationHandler 就不会陌生。

负载均衡在哪一侧？客户端 LB vs 服务端 LB

Dubbo 的 LoadBalance（Cluster 层）跑在 consumer 侧——这是「客户端负载均衡」。两种范式先分清：

客户端 LB（Dubbo / gRPC / SC LoadBalancer）服务端 LB（Nginx / 硬件 LB / 网关）
谁挑后端consumer 自己挑挡在后端前的代理挑
调用链路consumer → 直连选中的 provider（零额外跳）client → 代理 → provider（多一跳）
地址列表consumer 持有全部 provider 地址代理持有；client 只认一个虚拟地址

为什么 Dubbo 选客户端 LB：内部高 QPS 调用受不了一跳的延迟与代理瓶颈；而且 consumer 本来就握着全部地址（客户端服务发现，为了 failover/重试也得握着），挑一个是顺手的事；还能做 LeastActive、一致性哈希等需要调用上下文的精细策略。

本质：LB 永远在「调用方」那一侧

挑选需要两样东西——① 后端列表、② 调用上下文——只有调用方同时拥有；provider 连自己有哪些兄弟都不知道，没法挑。所以「服务端 LB」本质是「一个代理，它自己就是后端的调用方」。网关是后端的 caller、Nginx 是 upstream 的 caller、Dubbo consumer 是 provider 的 caller——所谓 server-side / client-side 只是它部署在哪儿，不是 LB 的本质。

真正该分的线：流量方向

南北向（外部 → 进入集群）东西向（集群内部 服务 ↔ 服务）
谁来干网关：Spring Cloud Gateway / Nginx / KongDubbo 客户端 LB（consumer 直连 provider）
职责鉴权 / 限流 / TLS / 路由 + LB在多 provider 间选一个 + failover / 重试

两者并存不互斥：网关管入口（南北），客户端 LB 管内部互调（东西）。你不会在两个内部 Dubbo 服务之间塞个网关——每跳都加一跳，高 QPS 下不可接受。顺带这也是 Dubbo3 Triple（HTTP/2，兼容 gRPC）的价值：让 Dubbo 服务能被标准 HTTP 网关（SCG / Envoy）甚至浏览器直接打到，把「内部 RPC」和「入口网关」打通（老 dubbo 协议是 TCP 二进制，网关不好转）。
