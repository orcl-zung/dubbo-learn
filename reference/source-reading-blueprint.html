一句话读法

拽住「一次调用:consumer 发起 → 跨网络 → provider 执行 → 结果返回」这根线,只读线碰到的东西。其余都是支线,先不读。线程驱动,不是特性驱动——这是攻读 Dubbo 唯一能活下来的姿势。

读法哲学(为什么是这形状)

  • 别做特性清单:Dubbo 15+ 模块组、几十个 jar,逐特性介绍 = 广度优先旅游,读到第 6 个前面全忘,且分不清谁跟"一次调用"有关。
  • 一根线:一次调用的往返。所有源码只按"它在这根线上吗"取舍。
  • 主路 vs 支线(不是"重要 / 不重要"):主路 = 回答"一次调用怎么完成"必须读的;支线 = 治理 / 性能 / 边缘特性,先划掉。
  • 往返镜像:同一批层 consumer 编码 / provider 解码互为倒影。按往返读,对称性自动 pop——provider 不单独开课,作为 consumer 的倒影来读。

最高 payoff 在 F

站 F(上线 Remoting)是 Dubbo 作为 RPC 框架最硬核、也是你目前最大的黑盒——调用到底怎么真的从 consumer 跨到 provider。概念最密,但也最值钱。这是大菜。

值得停下来品的设计(精妙榜)

不是一堆酷技巧,是少数几个深思想。按值得品的程度:

Top 3

设计为什么精妙在哪品
SPI 三件套:@SPI + @Adaptive + URL 总线元设计。换协议 / 注册中心 / LB / 序列化 = 换一个 URL 参数指向的扩展,框架代码零改动站 E / 第 4 课
Invoker / Invocation 单一抽象一切皆 Invoker、无限套娃;consumer 与 provider 用同一套抽象站 A / 图鉴
异步内核 + 同步外壳你写的同步 sayHello(),底层全 Future / 回调;RpcContext 把"当前调用"做成线程上下文贯穿全链站 F / J

强候选

  • Remoting 三层解耦(Exchange 请求-响应语义 / Transport 抽象 NIO / Serialize)——把"网络"层层脱到只剩"发请求等响应"。
  • 服务发现 接口级→应用级 + 迁移兼容——精妙在怎么平滑迁,不只是改了。
  • ProxyFactory 双向对称:getProxy(Invoker→代理) vs getInvoker(实例→Invoker),一个 SPI 两方向。
  • Triple = HTTP/2 + Protobuf + gRPC 兼容——打通"内部 RPC"与"标准 HTTP 网关"的战略设计。
